Task-010A：事項財務狀態＋提醒卡片收支摘要＋相關收支區塊整理

目前 Reminder Editor 已完成共通欄位統一與「分類 → 群組」UI 命名調整。

現在需要建立 Reminder 的財務狀態基礎，
並改善 Reminder Card 的財務資訊顯示。

本次任務只處理：

A. Reminder 財務狀態：無／待收／待付
B. Reminder Card 顯示待收／待付
C. Reminder Card 顯示已關聯 Finance Record 的收入／支出摘要
D. EditPage 的「相關收支」區塊位置上移
E. 「＋記帳」改名為「＋新增收支」

本次不要做：

- 確認收款
- 確認付款
- 自動建立 Income
- 自動建立 Expense
- 修改 Finance 首頁統計
- 修改 AI Parser
- 修改 Groq API
- 修改 /api/parse
- 大規模重構 App.tsx

一、Reminder 財務狀態資料

請依目前專案實際 Reminder 型別結構，
使用最小修改方式加入 optional 財務欄位。

需要支援：

financialStatus?: "none" | "receivable" | "payable"

expectedAmount?: number

financialDueDate?: string

語意：

none
＝ 無財務關聯

receivable
＝ 待收

payable
＝ 待付

重要：

所有新欄位必須保持 optional，
確保舊 Reminder 可以正常讀取。

不要進行破壞性 migration。

不要清除 LocalStorage。

不要修改既有 LocalStorage key。

目前 Reminder Card 可能只顯示：

請根據 Finance Records 的 sourceReminderId，
計算該 Reminder 的真正相關收支。

注意：

這裡只能計算 Finance Record。

不能計算：

Reminder.expectedAmount
Payment.amount
AirportTransfer.amount

────────────────

1 筆

────────────────

只有支出時：

例如關聯：

八、待收／待付與實際收支可以同時存在

Reminder 財務狀態與 Finance Record 是不同概念。

例如：

CI173 接機

Reminder：

待收 NT$ 1,000

Finance Records：

Expense 100
Expense 40

Card 可以顯示：

待收 NT$ 1,000

支出 - NT$ 140
2 筆

這是正確狀態。

因為：

1000 尚未收到，
但停車費與過路費已經實際支出。

────────────────

另一個例子：

信用卡繳費

Reminder：

待付 NT$ 23,560

如果尚無 Finance Record：

只顯示：

待付 NT$ 23,560

不能顯示：

支出 - NT$ 23,560

九、Finance Summary 計算規則

收入總額：

只加總該 Reminder 關聯的 Income Finance Records。

支出總額：

只加總該 Reminder 關聯的 Expense Finance Records。

必須依目前專案 Finance Record 的實際 type 欄位值判斷。

請先讀取現有型別與 Store，
不要自行假設 enum 大小寫。

amount 必須以 number 加總。

避免：

NaN
字串相加
重複計算
將 Reminder amount 加入 Finance Summary
將 expectedAmount 加入 Finance Summary
