Task-010A.4.3：統一新增事項與編輯事項的表單規格

目前發現新增事項頁與編輯事項頁仍然使用不同的表單呈現方式。

這造成：

1. 欄位順序不一致
2. 某些欄位只存在新增頁
3. 某些欄位只存在編輯頁
4. 時間 UI 不完全一致
5. 收支功能入口不一致
6. Type 專屬欄位排列不同
7. 未來新增欄位容易只修改其中一頁

本次目標：

讓「新增事項」與「編輯事項」使用相同的欄位規格、相同順序、相同元件與相同手機版布局。

重要：

本次不是重寫整個 App。
不要大幅重構 App.tsx。
優先共用現有欄位元件與 rendering logic。
不要修改資料 Store 核心結構。

────────────────────────

一、統一基本欄位順序

所有事項的新增與編輯頁，基本順序統一為：

事項類型

標題

群組

日期

時間

地點

提醒

備註

收支

提醒設定

行事曆

操作按鈕

新增頁與編輯頁必須遵守同一順序。

如果某個 Type 不需要某欄位，
可以不顯示該欄位，
但其他欄位相對順序不能改變。

────────────────────────

二、統一時間元件

新增頁與編輯頁必須使用同一套時間選擇方式：

[ 全天 ] [ 單一時間 ] [ 時間區間 ]

全天：

不顯示時間輸入。

單一時間：

時：分

時間區間：

開始時間
～
結束時間

不要讓新增頁與編輯頁使用不同的時間 UI。

不要恢復 HH:MM 文字輸入。

────────────────────────

三、統一群組欄位

新增與編輯頁都使用：

群組
[ 選擇群組 ▼ ]

＋ 新增群組

不要一頁叫「我的分類」，
另一頁叫「群組」。

統一顯示文字：

群組

資料仍使用目前既有 Category / Group 相容方式。

本次不要修改 Store 結構。

────────────────────────

四、Type 專屬欄位位置統一

Type 專屬欄位放在：

基本日期／時間／提醒之後
備註之前

例如 AirportTransfer：

事項類型
標題
群組
日期
時間
提醒

方向
航班
地區
車型
費用

備註

收支

提醒設定

行事曆

新增頁與編輯頁位置完全一致。

────────────────────────

五、AirportTransfer 清理舊財務 UI

AirportTransfer 保留：

方向
航班
地區
車型
費用

移除舊的：

財務狀態
[ 無 ] [ 待收 ] [ 待付 ]

預計金額

原因：

目前已經有統一收支系統。

費用只是接送工作的報價／費用資料。

不要因為填寫費用 1200，
就自動建立待收、收入或其他 Finance Record。

如果使用者需要建立款項，
透過統一：

收支
＋ 新增款項

建立：

待收
待付
收入
支出

────────────────────────

六、其他 Type 同樣檢查

檢查：

Course
Medical
AirportTransfer
Shopping
Payment
General

新增頁與編輯頁必須：

使用相同基本欄位順序
使用相同時間元件
使用相同群組元件
使用相同提醒元件
使用相同收支區塊

Type 專屬欄位可以不同，
但同一 Type 在新增與編輯時必須一致。

────────────────────────

七、Payment 特別規則

Payment 本身可以保留：

標題
截止日期
原始金額
帳戶／繳費方式
備註

但收支仍使用統一收支區塊。

不要再額外建立第二套：

財務狀態
待收／待付按鈕
預計金額

如果 Payment 原始金額是事項資訊，
保留原始金額。

但不要自動產生 Finance Record。

────────────────────────

八、Shopping 特別規則

Shopping 保留：

品項清單

例如：

牛奶
雞蛋
衛生紙
垃圾袋

新增與編輯頁使用相同的品項編輯元件。

收支仍走統一收支區塊。

不要因品項存在就自動建立支出。

────────────────────────

九、Medical 特別規則

Medical 保留：

醫院
科別

新增與編輯頁欄位順序一致。

醫療費用若需要記帳，
走統一收支區塊。

不要新增 Medical 專屬財務系統。

────────────────────────

十、Course 特別規則

Course 保留：

開始時間
結束時間
地點

但時間必須使用共用時間元件。

不要維持 Course 自己獨立的一套時間 UI。

────────────────────────

十一、共用規則

請檢查目前新增與編輯頁是否存在重複 JSX。

如果安全可行：

抽出最小必要的共用表單區塊，例如：

CommonReminderFields
TimeFields
ReminderFields
TypeSpecificFields
FinancialItemsSection

名稱可依現有專案風格調整。

但不要為了共用而大規模重構整個 App.tsx。

目標是：

同一欄位修改一次，
新增頁與編輯頁都能保持一致。

────────────────────────

十二、收支區塊規則

新增與編輯頁都顯示統一：

收支

目前沒有款項：
尚無收支款項

[ ＋ 新增款項 ]

已有款項：

待收 NT$ 3,500
日期
備註

待付 NT$ 2,000
日期
備註

收入 + NT$ 1,200
日期
備註

支出 - NT$ 100
日期
備註

新增頁建立的 Financial Item，
在 Reminder 儲存時一起保存。

不要因尚未取得 Reminder id 而直接提前建立孤立 Finance Record。

請沿用目前既有安全資料流程。

────────────────────────

十三、手機版 UI

請以手機寬度測試新增與編輯頁。

要求：

1. label 與 input 對齊方式一致
2. 不出現水平捲動
3. select 不超出容器
4. 日期欄位不超出容器
5. 時間選擇器不超出畫面
6. textarea 高度合理
7. 儲存／完成與取消按鈕不被 Bottom Navigation 遮住
8. 頁面底部保留安全間距
9. 不要讓瀏覽器底部工具列遮住主要按鈕

────────────────────────

十四、測試案例

測試 1：
新增 AirportTransfer。
確認欄位順序與編輯 AirportTransfer 完全一致。

測試 2：
新增 Course。
儲存後進入編輯。
確認時間模式與時間值一致。

測試 3：
新增 Medical。
確認醫院、科別在新增與編輯頁位置一致。

測試 4：
新增 Shopping。
確認品項清單新增與編輯方式一致。

測試 5：
新增 Payment。
確認沒有重複財務狀態 UI。

測試 6：
新增 General。
確認基本欄位與編輯頁一致。

測試 7：
AirportTransfer 費用輸入 1200。
儲存。
預期不自動建立待收或收入。

測試 8：
AirportTransfer 手動新增：
待收 1200。
儲存。
預期只有一筆待收 1200，不重複。

────────────────────────
