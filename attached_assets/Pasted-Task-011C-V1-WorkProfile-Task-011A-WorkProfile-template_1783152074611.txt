Task-011C：建立機場接送工作模板 V1，接入 WorkProfile

目前已完成：

Task-011A
- WorkProfile 資料模型
- 工作資料板管理
- templateType：
  general_work
  airport_transfer

Task-011B
- Reminder 支援 optional workProfileId
- 新增與編輯事項可以選擇工作資料板
- WorkProfile 刪除關聯保護
- 停用工作資料板相容處理

本次目標：

當 Reminder 選擇的 WorkProfile：

templateType = airport_transfer

顯示「機場接送工作資料」區塊。

本次只建立機場接送模板 V1。

重要原則：

1. WorkProfile 保存長期工作身份。
2. Reminder 保存單次工作事項。
3. Template Data 保存每一趟接送會改變的資料。
4. 收支繼續使用現有統一收支系統。
5. 不建立第二套金額、待收、待付、收入、支出系統。
6. 不刪除舊 AirportTransfer Type。
7. 不修改 AI Parser。
8. 不進行舊資料 migration。

────────────────────────

一、資料責任分離

WorkProfile 例如：

A 車行
templateType = airport_transfer

WorkProfile 本次不要保存：

單次航班
單次乘客
單次出發地
單次目的地
單次行李
單次金額

這些都是每一趟工作不同的資料。

Reminder 保存：

標題
群組
日期
時間
提醒
地點
workProfileId
備註
收支
提醒設定
行事曆

Airport Transfer Template Data 保存：

transferType
pickupLocation
stops
destination
flightNumber
passengerName
passengerPhone
passengerCount
luggage

────────────────────────

二、建立最小 Template Data

請依目前專案資料結構選擇最小安全方式。

建議：

templateData?: {
  airportTransfer?: {
    transferType?: "pickup" | "dropoff" | "charter" | "one_way"
    pickupLocation?: string
    stops?: string[]
    destination?: string
    flightNumber?: string
    passengerName?: string
    passengerPhone?: string
    passengerCount?: number
    luggage?: string
  }
}

如果目前專案已有 metadata、typeSpecificData 或類似結構，
優先評估沿用。

不要同時建立兩套 Template Data。

不要為每個 WorkProfile 複製一份模板 schema。

────────────────────────

三、模板載入條件

當 Reminder 選擇 WorkProfile：

A 車行
templateType = airport_transfer

顯示：

機場接送資料

如果選擇：

個人接案
templateType = general_work

不顯示機場接送資料。

模板顯示來源必須是：

Reminder.workProfileId
→ 找到 WorkProfile
→ 讀取 templateType
→ 決定 Template UI

不要使用工作名稱判斷。

禁止：

if name includes "車行"

或：

if name includes "接送"

只能依 templateType。

五、地點與路線的關係

共用表單仍保留：

地點

不要刪除共用 location。

機場接送模板另外保存：

出發地
中途停靠點
目的地

原因：

共用 location 可作為一般顯示、搜尋或主要集合地點。

路線資料則是機場接送專屬結構。

本次不要自動雙向同步：

location
pickupLocation
destination

避免資料互相覆蓋。

未來再決定主要地點策略。

七、收支保持完全獨立

機場接送模板中禁止新增：

待付

這些繼續使用目前 Reminder 的：

類型：

例如：

不要在 Template Data 再保存：

amount = 1200

避免再次產生重複金額問題。

────────────────────────

八、新增頁與編輯頁必須共用同一模板 UI

這部分非常重要。

目前專案已知歷史問題：

新增頁與編輯頁存在不同 rendering logic。

本次 Airport Transfer Template 不得建立：

CreateAirportTemplateFields

和：

EditAirportTemplateFields

兩套不同 JSX。

請建立單一可共用模板欄位元件或 rendering function。

例如概念：

AirportTransferTemplateFields

接受：

value
onChange
disabled

實際命名依專案風格。

新增頁與編輯頁必須使用同一套模板 UI。

十一、切換 WorkProfile 的安全規則

這部分必須小心。

例如 Reminder 原本：

A 車行
templateType = airport_transfer

已經填寫機場接送模板資料。

使用者切換：

個人接案
templateType = general_work

不要立即刪除 airportTransfer templateData。

先顯示確認：

切換工作後，目前的機場接送資料將不再顯示。

是否繼續？

[ 繼續 ] [ 取消 ]

V1 建議：

切換後保留舊 templateData，
但 UI 不顯示。

原因：

避免誤操作造成資料永久遺失。

如果之後重新切回 airport_transfer WorkProfile，
原模板資料可以重新顯示。

不要自動清空 Template Data。
