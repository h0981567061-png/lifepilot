請保留前一個 Task 已完成的 UI 與操作流程。

本次只修正：

AirportTransfer 與 Payment 的 Type 專屬金額，
和統一收支區塊之間的重複顯示與資料相容問題。

不要推翻前一個 Task。
不要重新設計整個收支區塊。
不要重構 Finance Store。
不要修改 AI Parser。
不要修改 Groq API。

────────────────

一、核心原則

同一個 Reminder 的「預計款項」只能有一個主要編輯位置。

例如 AirportTransfer：

接送費用 1200

如果這個 1200 的語意是預計向客戶收取，
Editor 不應同時出現：

接送費用：1200

以及：

待收
金額：1200

兩個可以分別修改的欄位。

Payment 同理。

不要讓同一筆金額存在兩個互相可能不同步的 UI 輸入位置。

────────────────

二、AirportTransfer 相容方式

保留 AirportTransfer 既有 amount 資料欄位，
不要刪除欄位，
不要破壞舊資料。

Editor UI：

接送資訊區保留：

接送類型
航班號碼
地區
車型
備註

如果 AirportTransfer.amount 是接送工作的預計款項，
不要再於接送資訊區顯示獨立可編輯的：

接送費用

改由統一「收支」區塊負責顯示與編輯預計款項。

例如舊資料：

type = AirportTransfer
amount = 1200
financialStatus 尚未設定

Editor 相容顯示：

收支

預計款項：
待收

金額：
1200

但不要建立 Income Finance Record。

不要將 1200 計入正式收入。

────────────────

三、AirportTransfer 儲存相容

使用者在統一收支區塊修改：

待收
1200

儲存時請維持目前資料相容策略。

至少確保：

financialStatus = receivable
expectedAmount = 1200

如果現有 AirportTransfer UI 或其他舊程式仍依賴 amount，
可以同步維護 amount = 1200。

但必須只有一個 UI 編輯來源。

不要讓：

amount = 1200

expectedAmount = 1500

產生 UI 修改後不同步的狀況。

請使用最小修改方式處理。

────────────────

四、Payment 相容方式

保留 Payment 既有：

amount
dueDate

不要刪除。

例如：

信用卡繳費
amount = 23560
dueDate = 2026/08/05

Editor 統一收支區塊顯示：

待付

金額：
23560

付款期限：
2026/08/05

不要另外再出現第二組可編輯的：

付款金額
待付金額

或：

截止日期
付款期限

如果兩者實際代表相同語意，
UI 只保留一個主要編輯位置。

底層可以為舊資料相容而同步欄位，
但 UI 不要重複。

────────────────

六、同一事項可以同時有預計款項與實際收支

例如接送機：

待收 NT$ 1,200

同時已有：

停車費支出 NT$ 100
過路費支出 NT$ 40

這是合法狀態。

UI 摘要應可顯示：

待收 NT$ 1,200

實際收支
支出 - NT$ 140
2 筆

不要把「待收」與「支出」做成互相覆蓋的資料。

如果前一版的五選一 UI 會造成選擇「支出」後清除「待收」，
請修正這個行為。

UI 可以用同一個收支區塊，
但 Reminder 預計款項與 Finance Record 實際收支必須能同時存在。

────────────────

七、收入／支出按鈕行為

如果已有待收或待付狀態，
使用者新增實際收入或支出時：

不要清除原本 financialStatus。

例如：

Reminder：
待收 1200

使用者新增：
支出 100

結果必須保持：

Reminder：
待收 1200

Finance Record：
Expense 100

不能變成：

Reminder financialStatus = none

也不能因新增 Expense 而刪除待收資料。
