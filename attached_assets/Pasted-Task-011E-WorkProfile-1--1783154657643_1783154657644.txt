Task-011E：重整工作建立／編輯架構，整合工作資料與自訂欄位

目前已完成 WorkProfile 與工作管理功能，但工作資料流程需要重新調整。

本次正式架構原則：

1. 工作由使用者自行建立。
2. 新增工作與工作資料必須在同一頁完成。
3. 不再使用「新增工作 → 儲存 → 下一頁填工作資料」流程。
4. 一般工作顯示通用工作欄位。
5. 機場接送是目前唯一的特殊工作格式。
6. 所有工作都可以新增自訂欄位。
7. 「啟用此工作」必須放在資料欄位之後。
8. 一般 UI 不顯示「工作模板」等技術名稱。
9. 不使用個人資料作為 placeholder 範例。

────────────────────────

一、移除獨立工作資料頁流程

目前如果存在：

我的工作
→ 建立工作
→ 儲存
→ 工作資料
→ 再填資料

請停止使用這個流程。

修改為：

我的工作
→ 新增工作
→ 同一頁填寫工作名稱與工作資料
→ 設定是否啟用
→ 儲存

編輯工作也使用相同表單結構。

不要建立兩套不同的新增與編輯 UI。

建議共用：

WorkProfileForm

或依專案既有命名方式建立共用元件。

五、機場接送特殊格式的識別

重要：

一般使用者 UI 不顯示：

templateType
capability
schema
airport_transfer

但內部仍需有安全的識別方式。

請檢查目前 WorkProfile 是否仍保留：

templateType

或：

capability

或其他既有安全識別欄位。

如果既有資料已經有：

templateType = airport_transfer

可以繼續用於內部 UI renderer 判斷。

但不要在表單中顯示模板選擇器。

禁止使用：

work.name.includes("機場")
work.name.includes("接送")
work.name === "機場接送"

作為正式能力判斷。

工作名稱只是使用者輸入的顯示名稱。

六、新建立工作如何決定格式

請先檢查目前專案是否已有安全的機場接送工作建立入口或既有 capability 指派方式。

本 Task 原則：

一般新增工作預設使用 general_work 內部格式。

既有 airport_transfer WorkProfile 保留 airport_transfer 能力。

不要因使用者輸入「機場接送」四個字，就永久自動修改 capability。

如果目前沒有安全的特殊工作建立入口：

本 Task 不要用 name.includes() 補洞。

請保留一般工作建立流程，
並回報目前 airport_transfer capability 是如何建立或保留的。

不要自行發明 AI 自動分類流程。

七、自訂欄位資料模型

所有工作都支援：

＋新增自訂欄位

建議資料結構：

customFields: [
  {
    id: string,
    label: string,
    value: string
  }
]

實際請依現有資料架構整合。

要求：

1. 每個自訂欄位有穩定 id。
2. label 可以編輯。
3. value 可以編輯。
4. 可以新增。
5. 可以刪除。
6. 編輯工作時可以修改。
7. 重新整理後仍存在。
8. 不因工作停用而刪除。
9. 不因固定欄位切換顯示而刪除。

十、啟用狀態位置

「啟用此工作」必須放在：

所有固定資料欄位
＋
所有自訂欄位

之後。

順序必須是：

工作名稱
↓
固定資料
↓
自訂欄位
↓
啟用此工作
↓
新增／儲存

不要把啟用開關放在資料卡前面。

enabled 只代表：

這份工作目前是否使用中。

enabled 不負責：

決定是否顯示工作資料欄位。

不要使用：

enabled === true 才顯示 profileData form

工作資料表單在建立／編輯時都可以填寫。
