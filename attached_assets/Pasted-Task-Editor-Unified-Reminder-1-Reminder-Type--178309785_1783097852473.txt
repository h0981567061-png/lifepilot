Task-Editor-Unified：統一所有 Reminder 編輯表單＋分類命名改為群組

目前專案存在兩個需要一起整理的問題：

問題 1：
不同 Reminder Type 使用不同的編輯表單，
導致新增共通功能時容易漏掉部分類型。

例如目前可能出現：

- Medical 有時間設定
- Course 有開始與結束時間
- AirportTransfer 有單一時間
- Shopping 缺少時間功能
- Payment 使用不同的日期與時間邏輯
- 某些類型有提醒功能，某些類型容易漏掉
- 某些類型有地點，某些類型沒有

問題 2：
目前 UI 使用：

- 事項類型
- 我的分類
- 分類管理

容易讓使用者誤解「事項類型」與「我的分類」是相同概念。

本次任務目標：

A. 統一所有 Reminder 編輯表單的共通欄位
B. 統一所有 Reminder 的時間操作
C. 保留各 Type 的專屬欄位
D. 將 UI 中的「我的分類」概念改名為「群組」
E. 不修改底層 Category Store 與現有資料格式

────────────────

一、核心架構原則

所有 Reminder Type 必須使用一致的基本編輯結構。

不要再讓：

Medical
Course
Shopping
Payment
AirportTransfer
General

各自擁有彼此完全不同的基本表單。

所有 Reminder Type 都必須具有以下共通能力：

1. 事項類型
2. 標題
3. 群組
4. 日期
5. 時間模式
6. 地點
7. 提醒設定
8. 行事曆開關
9. 備註

重要：

共通功能不能因 Reminder Type 不同而消失。

例如：

- Shopping 也可以設定時間
- Payment 也可以設定地點
- Medical 也可以使用時間區間
- AirportTransfer 也必須有提醒設定
- Course 也必須有群組與備註
- General 也必須有完整共通欄位

────────────────

所有 Reminder Editor 基本順序統一為：

標題

然後才顯示該 Type 的專屬欄位。

如果目前專案 UI 結構更適合將專屬欄位放在「地點」前後，
可以依現有版面微調，

但所有 Type 的共通欄位順序必須盡量一致。

三、時間功能統一

所有 Reminder Type 都支援三種時間模式：

1. 全天
2. 單一時間
3. 時間區間

不要限制某個 Type 只能使用某種模式。

────────────────

四、全天模式

選擇：

全天

行為：

- 不顯示時、分選擇器
- Reminder 保留日期
- UI 顯示為全天事項

例如：

購買生活用品
2026/07/18
全天

────────────────

五、單一時間模式

選擇：

單一時間

顯示：

時：分

例如：

10 : 00

請使用目前已有的時間選擇器 UI。

不要使用純文字輸入 HH:MM。

例如 Medical：

帶媽媽看骨科
2026/07/15
10:00

────────────────

六、時間區間模式

選擇：

時間區間

顯示：

開始時間
時：分

至

結束時間
時：分

例如 Course：

小孩英文課
2026/07/16
19:00－21:00

所有 Type 都可以使用時間區間。

例如 Medical 也可能有：

09:00－12:00

Shopping 也可能有：

18:00－20:00

不要限制。

七、時間模式切換

請檢查現有 Reminder Data。

優先相容目前已有：

allDay
time
startTime
endTime

或專案目前實際使用的欄位。

不要進行破壞性 migration。

如果舊資料：

有 startTime + endTime
→ Editor 顯示為時間區間

→ Editor 顯示為全天

如果欄位不存在，
使用 optional 欄位方式保持相容。

九、Type 專屬欄位

所有 Type 先有共通欄位，
再額外增加自己的專屬欄位。

Medical：

- 醫院
- 科別

Course：

- 保留目前已有的課程專屬資料欄位
- 不要刪除現有 Course Data

Shopping：

- 品項清單
- 新增品項
- 刪除品項

Payment：

- 截止日期
- 金額
- 帳戶或付款方式

AirportTransfer：

- 接送方向
- 航班
- 地區
- 車型
- 接送費用

General：

- 不需要額外專屬欄位

重要：

專屬欄位是「增加」，
不是取代共通欄位。

────────────────

十、AirportTransfer 金額語意

AirportTransfer 的 amount 在 UI 顯示為：

接送費用

不要顯示模糊的：

金額

AirportTransfer.amount 代表：

該接送事項的預計車資或接送費用。

它不是 Finance Income Record。
也不是 Finance Expense Record。

本次不要自動建立收入或支出。

────────────────

十一、Payment 與 Finance 保持分離

Payment Reminder 代表：

需要付款的提醒事項。

例如：

信用卡 8/5 前繳 23560 元

它不等於已經完成一筆 Expense Record。

本次不要因 Payment Reminder 有 amount，
就自動建立 Expense。

Reminder 與 Finance Record 必須保持資料概念分離。
